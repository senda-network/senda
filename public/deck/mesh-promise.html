Current gap

Routable is not the same as pleasant.

Example from the live mesh: Gemma-27B on a MacBook Air over ~140ms WAN sits around 1–3 tok/s. The serve is genuine. Showing it as everyday chat is the mistake.

Dialable ≠ chat-viable Publish the measurement, don’t dress it up

Rules we are locking

How routing and UI should behave

  • No silent model swap — if you pick Gemma, you get Gemma (or an explicit, labeled backup choice later)
  • Say who served it — mesh peer vs cloud backup is not a hidden detail
  • Show speed before send — best-peer TTFT / tok/s in the picker, not only on /status
  • Don’t elect unfit hosts — undersized / ~1 tok/s capacity shouldn’t look “ready”

What we’re adding · for node runners

Fit the model to the machine

  • Election prefers peers that can actually run the weights well
  • “Loading” on undersized VRAM stops counting as ready supply
  • Credits and reputation weight healthy sessions, not just presence
  • Status stays the public proof of what you contribute

What we’re adding · for the marketplace

Grow mesh share without faking quality

  • Track mesh_share_pct — traffic actually served by peers
  • Report daily-driver UX separately from capacity demos
  • Cloud backup later, only as a labeled uptime option with billing
  • Paid API path: customers pay; peers earn on verified sessions

Out of bounds

Things we are not shipping

  • Auto-route capacity SLA misses to OpenRouter under the same model name
  • Green “online” for hosts stuck at ~1 tok/s with no warning
  • Counting bad routes as mesh-share wins
  • Keep the slow serve if the user opted in with the numbers visible
